# Dropbox API Experiment Plan

## Current behavior

The current reader is a customized PDF.js viewer hosted from GitHub Pages.
Chapter links are generated by a separate staff tool, and each patron link opens
`web/viewer.html` with a proxied PDF URL in the `file` query parameter.

The viewer currently:

- reads a token from the URL or from the proxied `file` URL,
- hides print and download buttons unless the token allows downloads,
- blocks common save/print keyboard shortcuts,
- redirects the patron back inside the selected chapter if they navigate outside
  the token's start and end pages.

Those controls are useful for normal patrons, but they are browser-side controls.
They should not be treated as the security boundary.

## Important security rule

Dropbox API credentials must never be placed in GitHub Pages JavaScript.

The reader runs in the patron's browser. Any Dropbox access token, app secret,
or reusable Dropbox file URL placed in browser code can be found by a technical
patron through developer tools.

The experiment should therefore use a server-side proxy, most likely a
Cloudflare Worker, with Dropbox credentials stored as Worker secrets.

## Recommended architecture

1. The staff generator creates a chapter link with a signed token.
2. The token identifies:
   - the Dropbox file ID or approved Dropbox path,
   - the allowed chapter start page,
   - the allowed chapter end page,
   - download policy.
3. The patron opens the static PDF.js reader.
4. The reader asks the Worker for a short-lived reader session.
5. PDF.js requests content from the Worker, not from Dropbox directly.
6. The Worker validates the token and reader session before contacting Dropbox.
7. The Worker uses the Dropbox API to fetch only what the current design allows.

This keeps PDF.js and streams the PDF through the Worker with HTTP Range support.
It should improve initial load time for large PDFs when the source PDF and
network support partial reads.

Tradeoff: the browser still receives PDF bytes. A technical patron may still be
able to save the allowed chapter PDF bytes. The current goal is not to make a
PDF impossible to save; it is to avoid exposing Dropbox credentials or the full
source PDF, while making copied URLs hard to reuse outside the approved reader
flow.

## First proof of concept

Start with Track A because it is closest to the existing working reader:

1. Keep the existing PDF.js viewer.
2. Move Dropbox access behind a Worker.
3. Keep Dropbox links private or set downloads off in Dropbox.
4. Store Dropbox credentials as Worker secrets.
5. Add Range-aware proxying in the Worker.
6. Confirm the reader can load a large PDF faster.
7. Confirm no Dropbox shared link or Dropbox API token appears in browser
   network requests.

## Lab Worker scaffold

The `worker/` folder contains the first scaffold:

- `worker/src/index.js` signs chapter tokens and validates them before touching
  Dropbox.
- `worker/src/index.js` proxies Dropbox PDF bytes through the Dropbox API.
- `worker/src/index.js` supports Dropbox refresh-token secrets so the Worker can
  keep operating after short-lived Dropbox access tokens expire.
- `worker/wrangler.toml` defines a Cloudflare Worker that can be deployed later.

The first scaffold accepts Dropbox API file references such as `/Folder/Book.pdf`
or `id:...`, and can also resolve Dropbox shared links server-side when the
Dropbox app has `sharing.read`.

The lab now keeps the simpler chapter-only PDF approach. The Worker also checks
that patron PDF requests come from the approved PDF.js reader origin and include
a short-lived reader session, so copying the raw Worker URL should not be useful
by itself.
